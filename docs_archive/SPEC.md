## 1. 起源

### 1.1 问题

**超复杂系统的结构无法在构建之前被完全认知。** 传统软件工程建立在"结构可在构建前设计"的假设上。这个假设对庞杂系统成立。对复杂系统不成立——组件之间的交互在运行时涌现，且随时间演化。

在超复杂系统开发中：
- 改一个节点 → 系统全局的耦合关系可能发生非局部变化
- 这种变化不是立即可见的——如果变更影响的是一条异步或周期执行路径，它可能在延迟之后才暴露
- 暴露时不知道是哪个更改导致的——因果链跨越了太多中间节点，且传播路径不沿文件目录结构展开
- 目录树 + 文件逐页阅读（"笔记本范式"）无法承载这种非线性的、多维的、时间不对称的关系网络

### 1.2 根因

**人脑的线性摄入带宽不足以承载超复杂系统的并行关系结构。**

- 目录树 = 位置作为组织原则 = 一次一个文件 = 翻页断上下文
- 系统的耦合不因你看不到而消失——它存在于调用图、数据流图、并发执行图中，这些图交叉叠加，任何单一的线性表示都会割裂它们
- 工具的组织原则反向训练使用者的认知结构：目录树训练人把代码当孤立的文件看，而不是当成一张连通的图

### 1.3 方向

一个系统无关的观测工具，将代码库的结构从**序列表示（文件/目录）转换为空间表示（节点/边/拓扑）**，让人类的视觉-空间并行系统来处理系统结构。

核心洞察：**你不应该翻代码去"找"关系。你应该站在图里，"看见"关系。**

## 2. 核心定义

### 2.1 本质

全息观测站是一个**代码库的实时可交互依赖拓扑图生成器，同时服务于人和 LLM**。

```
输入：一个目录路径（零配置）
输出：一张可以站进去的全息图 + 一份 LLM 可消费的结构化图数据
```

在当前及未来的开发范式中，LLM 已逐渐成为代码操作的主力。全息观测站不是单纯的"人的决策辅助"，而是**人 + LLM 之间的共享认知界面**：人通过空间视图做判断，LLM 通过结构化图数据做精确的波及计算。二者消费同一张图的两种投影。

### 2.2 交付形态：Agent 指挥台

形态不是 IDE 扩展，是**独立的桌面子应用**。在 vibe coding 范式下，代码的读写主力是 LLM，人需要的只有两样东西：

1. **跟 Agent 对话的地方** — 描述意图、确认方案、审查结果
2. **看系统结构的地方** — 全息星图，感知代码的拓扑形态

屏幕布局：
```
┌─────────────────────────────────────────────────────┐
│                                                     │
│              全息星图（桌面背景，始终可见）              │
│         代码库的完整拓扑 · 实时响应变更                │
│                                                     │
│          ┌──────────┐    ┌──────────┐               │
│          │ README.md│    │ config   │               │
│          │ [预览模式]│    │ [编辑模式]│               │
│          └──────────┘    └──────────┘               │
│                                                     │
├─────────────────────────────────────────────────────┤
│  终端嵌入（xterm.js）                                │
│  $ claude                                           │
│  > 改 handle_request … 波及 47 节点，3 个有延迟……    │
└─────────────────────────────────────────────────────┘
```

图是桌面背景，占屏幕 70-80%，始终可见。文件窗口浮在图上——可拖拽、可缩放、不挡关心的节点。终端占底部 20-30%。Claude Code 改了文件 → 文件系统事件 → 图自动刷新。图和 Agent 在同一窗口内。

### 2.3 核心原则

1. **观测站必须比被观测的系统简单一个数量级**
2. **系统无关** — 不锚定任何具体语言、具体领域
3. **关系是唯一的一等公民** — 文件、目录、模块退化为"节点的一种属性"
4. **代码是图唯一的真相来源** — 图从代码自动提取，不依赖人工标注
5. **只描述，不解释** — 观测站不关心函数干了什么，只关心它连接了谁
6. **同一张图，两种消费** — 图的 JSON 数据是人和 LLM 的共同接口；人通过空间投影看，LLM 通过结构化数据读

### 2.4 Agent 无关性

全息指挥台不绑任何特定 Agent。任何 vibe coding 开发者都能接上自己喜欢的 Agent。

**通道 1：MCP Server（Agent 主动查询）** — 7 工具
```
hologram_neighbors(node_id)        → 一阶邻接（按边类型分组）
hologram_impact(node_id, depth)    → 波及分析（BFS 分层 + 延迟标注）
hologram_path(from_id, to_id)      → 两点间所有路径
hologram_history(node_id)          → 该节点的决策历史
hologram_community(node_id)        → 所属社区 + 兄弟节点
hologram_delayed()                 → 所有含时间延迟边的节点
hologram_changes()                 → 上次变更的回看标记（命中/漏判/过估）
```

**通道 2：CLI 工具（兜底方案）**
```
$ hologram neighbors handle_request
$ hologram impact handle_request --depth 3
$ hologram path handle_request health_checker
```
任何能调 bash 的 Agent 都能用。MCP 和 CLI 底层是同一套 Python 函数——只是暴露方式不同。

**反向通道（星图 → Agent）**
```
星图上的交互事件 → 写入终端标准输入 → Agent 感知：
  "用户正在查看 handle_request，一阶邻接 16 节点，含 3 个延迟"
```

## 3. 图的骨架：三种节点 × 三种边

这是整个系统最重要的设计决策。

### 3.1 节点类型

任何系统中"可以命名的事物"只属于以下三类：

**符号节点（Symbol Node）** — 代码中定义的、命名的、可引用的结构。
```
kind: function | class | module | constant | interface | variable
属性: name, location, language, kind, scope
```

**介质节点（Medium Node）** — 代码之外，代码会碰的持久化或共享资源的抽象。
```
kind: file | database | queue | cache | shared_memory | network
属性: name, kind, uri
```

**时间节点（Temporal Node）** — "什么时候执行"的结构。不标注这类节点，就看不到因果延迟。
```
kind: thread | scheduler | event_loop | trigger
属性: name, kind, interval_sec, is_daemon
```

### 3.2 边类型

任何两个节点之间的连接关系只属于以下三类：

**结构边（Structural Edge）** — 符号之间的静态代码关系。
```
direction: call | inherit | implement | import | reference | instantiate
```

**数据边（Data Edge）** — 符号与介质之间的 I/O 关系。
```
direction: read | write | subscribe
```

**时间边（Temporal Edge）** — 符号与时间节点之间的执行关系。
```
direction: executes_on | triggers | blocks
属性: temporal_delay_sec
```

### 3.3 闭合性

```
你在代码里定义了一个东西   → 符号节点
它引用了另一个东西         → 结构边
它读写了一个外部资源       → 介质节点 + 数据边
它运行在一个并发结构上     → 时间节点 + 时间边
```

任何语言、任何系统里的任何关系，都是这六种元素的组合。不需要更多类型，也不能更少。

### 3.4 跨语言适配器

当前已实现：**Python**（AST 解析）和 **TypeScript/JavaScript**（正则模式匹配）。
每个语言一个适配器，实现同一接口：

```python
class LanguageAdapter(Protocol):
    language: str

    def accepts(file_path: str) -> bool
    def extract_symbols(paths: list[str]) -> list[SymbolNode]
    def extract_structural_edges(nodes: list[SymbolNode]) -> list[StructuralEdge]
    def extract_media(paths: list[str]) -> list[MediumNode]
    def extract_data_edges(nodes: list[SymbolNode], media: list[MediumNode]) -> list[DataEdge]
    def extract_temporal_nodes(paths: list[str]) -> list[TemporalNode]
    def extract_temporal_edges(nodes: list[SymbolNode], temporal: list[TemporalNode]) -> list[TemporalEdge]
```

适配器是无状态的。每个适配器只负责自己的语言，不关心其他适配器的存在。

## 4. 构建流水线

```
阶段 0: 文件发现 — 扫描目录 → 识别文件类型 → 分发给对应适配器
阶段 1: 符号提取（并行）— 每个适配器独立运行 → 符号节点 + 结构边
阶段 2: 介质发现（并行）— 扫描 I/O 操作 → 介质节点 + 数据边
阶段 3: 时间标注（并行）— 扫描并发结构 → 时间节点 + 时间边
阶段 4: 图合并 — 去重 → 统一 ID → 验证 → 完整图
阶段 5: 社区聚类 — Leiden 算法对合并后的图做自动分区 → 层级社区结构
```

关键设计点：
- 介质发现基于模式匹配（不需要真正执行代码），维护一个已知库的 I/O 签名映射表
- 时间标注匹配常见并发模式（threading.Thread、Timer、asyncio、while+sleep）
- 图合并时介质节点跨语言合并：Python 和 TypeScript 可能读写同一个文件
- 所有边权重的默认值见 §5，随验证积累可调整

## 5. 社区聚类与折叠式拓扑

### 5.1 毛线球问题的解

把所有节点平铺在屏幕上 = 毛线球。解决方式：**分辨率由用户控制，默认从宏观层级开始。**

```
初始状态：      Level 2 社区（职责域），只显示社区节点
点击一个社区：  展开 → Level 1 子社区
点击一个子社区： 展开 → Level 0 具体节点
点击一个节点：  展开 → 一阶邻接
```

### 5.2 算法

Leiden（比 Louvain 收敛更好），层级：Level 0（单节点）→ Level 1（基本面）→ Level 2（职责域）→ Level 3（整个项目，1 个根社区）。

边权重默认设置：
- 结构边调用 → 1.0
- 数据边同介质读写 → 1.5（共享状态产生强耦合）
- 时间边同线程/调度器 → 1.0
- 同一文件内 → 额外 +0.5（物理内聚）

这些权重是初始默认值，随验证积累调整。

### 5.3 社区呈现

折叠态：社区名（介数中心性最高的若干符号名拼接）、节点数/边数、内部耦合密度（颜色深浅映射）、外部连接数（边粗细映射）。

跨社区边默认显示，但只在当前焦点社区的出入边高亮，其余半透明。

## 6. 交互层设计

### 6.1 设计目标

- 给视觉-空间系统喂数据（不给序列通道喂数据）
- 操作单调：点击（展开/聚焦）、拖拽（平移）、滚轮（缩放）
- 状态记忆：在某节点展开的状态在浏览其他节点时不丢失

### 6.2 三个核心功能

任何开发者在改代码之前，潜意识里都在问三个问题：

**功能 1：看邻居 — "这个东西连着谁？"**
- 触发：单击节点
- 频率：最高（80%+ 的交互）
- 响应：一阶邻接节点和边高亮，按类型着色，其余降不透明度；侧边栏出详情

**功能 2：看波及 — "动这个东西会波及多远？"**
- 触发：右键 → Pie Menu "波及"
- 频率：中频
- 响应：沿所有类型边 BFS，按距离分层着色（震中/一阶/二阶/三阶）；时间边连上的节点加延迟标签

**功能 3：看路径 — "这两个不相关的东西怎么连上的？"**
- 触发：A 设锚点 → B 触发路径
- 频率：低频
- 响应：显示 A→B 所有可达路径，非路径节点消隐

**功能 4：变更回看 — "改了之后，实际跟预判一致吗？"**
- 触发：快照比较后图上自动叠加标记
- 频率：低频（每次变更周期结束）
- 响应：红(命中)/黄(漏判)/绿(过估) 节点着色 + 统计面板

### 6.3 从 Blender 借鉴的交互范式

| 模式 | Blender 原版 | 全息映射 | 解决什么 |
|---|---|---|---|
| Proportional Editing | 按 O，拖拽顶点，周围跟着动 | 波及环：渐变圆圈标影响半径，滚轮调半径 | 波及不用表格，用视野感知 |
| Local View | 按 `/`，只看选中物体 | 双击节点进入聚焦模式，其余全消，ESC 回全局 | 噪音消除 |
| Pie Menu | 右键饼形菜单 | 四个核心操作，四个方向，肌肉记忆 0.1 秒 | 统一入口 |
| 3D Cursor | 放光标作为参考点 | 路径锚点：右键设起点 → 右键设终点 → 自动算路径 | 路径搜索的自然交互 |
| Outliner | 大纲面板树形列表 | 社区树面板：按耦合聚类组织，替代目录树 | 结构导航 |

### 6.4 视图模式

- **社区视图（默认）** — Level 2 社区 + 跨社区边。理解宏观结构
- **展开视图** — 焦点下的 Level 0-1 + 所有边类型。理解局部耦合
- **影响追踪视图** — BFS 波及分层着色。回答"改了会波及什么"
- **路径视图** — 仅显示 A→B 路径。回答"怎么连上的"

### 6.5 边的视觉编码

| 边类型 | 颜色 | 线型 |
|---|---|---|
| 结构边 | 蓝色 | 实线 |
| 数据边（读）| 绿色 | 虚线 |
| 数据边（写）| 红色 | 虚线 |
| 时间边 | 橙色 | 点线 |
| 跨社区边 | 灰色半透明 | 粗线 |

### 6.6 浮动文件窗口

星图为底，文件以 `position:absolute` 浮动在星图上方。可拖拽标题栏、缩放右下角、最小化（缩成社区附近的标记点）、关闭。支持同时打开多个文件散布在星图不同位置。Diff 模式：点击 `[⟺]` 翻面显示 git diff 并排对比。

| 文件类型 | 默认 | 可切换 |
|---|---|---|
| Markdown (.md) | 预览（渲染 HTML）| 编辑（Monaco）|
| 代码文件 | 编辑（Monaco）| Diff (git diff 视图) |
| 配置文件 | 编辑 | Diff |
| 图片 | 预览 | — |

## 7. 变更影响分析

### 7.1 三段式工作流

**阶段 1：提案前 — LLM 先看 (Pre-proposal)**
```
LLM 接收指令 "改 A"
  → 读取全息图的波及子图（结构化 JSON）
  → 产出方案并标注波及：
    "方案 A（直接改）：波及 47 节点，含 3 个延迟
     方案 B（换切入点）：波及 5 节点
     建议方案 B。你确认？"
```

全息图此时是 LLM 的精确查询工具，而非靠注意力机制在代码全文上猜。

**阶段 2：裁决 — 人看图确认 (Judgment)**
```
人打开全息图，焦点在 A，一眼扫过去确认波及范围跟 LLM 说的对得上
→ "通过，按方案 B 改"
```

**阶段 3：事后 — 共同回溯 (Retrospective)**
```
变更 commit 后，图上叠加变更标记：
  红色 — 实际改了，在波及预测内（正确预判）
  黄色 — 实际改了，不在波及预测里（漏判 → 波及模型有盲区）
  绿色 — 在波及预测里，但实际未改动（可能过度估计）

人和 LLM 共同看图回顾：
  "这次的漏判节点是 cache_sync_worker。
   它跟变更节点之间有一条 4h 延迟的时间边。
   要不要把这条新边加到认知里？"
```

### 7.2 LLM 作为结构化数据消费者

LLM 消费全息图的方式不是"看图"，是**读结构化数据**：

```
查询类型：
  · 返回节点 X 的一阶邻接（按边类型分组）
  · 返回节点 X 的波及子图（BFS N 层）
  · 返回节点 A 到节点 B 的所有路径
  · 返回上一个 commit 的变更标记
  · 返回当前图中含时间延迟边的所有节点

返回格式：JSON，字段包括节点 ID、名称、位置、边类型、传播距离、延迟信息。
```

### 7.3 实现方式

**波及预测 (Pre-commit)**：从变更节点出发，沿所有类型边 BFS，按距离分层标注，标注经由时间边的延迟节点。输出波及节点集合 + LLM 可消费的 JSON 子图。

**实际变更比对 (Post-commit)**：diff 两张图 → 识别实际变更节点 → 如果存在波及预测则计算命中/漏判/过估 → 上调漏判节点与变更节点之间的路径边权重。

与 git 集成但不重新实现 git：`git diff --name-only` 识别变更文件 → 增量重新分析 → 对比新旧图。

## 8. 决策时间轴

星图展示代码的结构。git 记录文件的变更。但项目开发中还有第三层信息：**人 + Agent 的决策史**——今天改了什么、为什么改、讨论后决定不改什么。这些不在 git 里，在对话历史里。

决策时间轴将这部分信息从对话流中提取出来，与星图和 git commit 对齐，挂载在星图边缘。

### 8.1 形态

```
星图右侧边缘，一条竖的时间轴：

今天 14:32  — 改了 handle_request，波及 5 节点，无延迟
今天 11:07  — 加了 validate_session，新建 8 条边
昨天 18:22  — 讨论：是否加第 6 个冲动源 → 决定不加
昨天 09:15  — 重构了 consolidation 模块，社区从 3 裂成 5
```

每一条是一个决策锚点。点击锚点 → 星图切换到当时的快照 + 显示当时的对话摘要。

### 8.2 三层对齐

```
git commit   → 该 commit 对应的星图快照
对话片段     → 该变更的讨论上下文（"为什么这么做"）
波及数据     → 该变更的实际波及范围（事后校验）
```

git 告诉你"什么变了"，决策时间轴告诉你"为什么变"和"波及了什么"。

## 9. 技术栈与项目结构

### 9.1 技术栈

| 维度 | 选择 | 理由 |
|---|---|---|
| **后端语言** | Python | 标准库 AST；NetworkX / leidenalg 图生态 |
| **桌面壳** | Tauri 2.x | Rust 壳，~5MB vs Electron ~120MB；跨平台 |
| **全息星图渲染** | Cytoscape.js v3 + cose-bilkent | 专门为交互式图可视化设计 |
| **终端嵌入** | xterm.js v5 + FitAddon | VS Code 同款终端底层 |
| **代码编辑** | Monaco Editor | VSCode 内核，轻量 |
| **Markdown 渲染** | marked.js | 轻量 MD → HTML |
| **前端构建** | Vite 5 + TypeScript 5 | ESM 原生；Tauri 官方推荐 |
| **图存储** | 内存 (NetworkX) | 初版规模不过百万节点，纯内存零 I/O |
| **社区发现** | leidenalg + igraph | 算法质量优于 NetworkX 内置 Louvain |
| **文件监听** | watchdog | 代码变更 → 自动触发重分析 |
| **前后端通信** | Tauri event system | 桌面壳内通信，不走 HTTP，不暴露端口 |

## 10. 开放议题

这些不是"待解决 bug"。它们是研究路线——这个项目的本质是在探索一个问题：**超复杂系统的结构怎么被人理解。**

| 议题 | 当前状态 |
|---|---|
| 介质节点的识别策略：纯规则 vs 混合 LLM 辅助？ | 纯规则，收集覆盖率数据后评估 |
| 社区命名（自动生成的标签）可读性差？ | 介数中心性符号名拼接；后续可 LLM 辅助命名 |
| 大项目（100 万行以上）的图生成性能？ | 增量缓存 + 并行适配器；先收集性能数据 |
| 力导向图在 1000+ 可见节点时性能？ | 社区折叠设计保证同屏节点数可控 |
| 跨语言介质节点如何合并？ | 单语言先跑通，混合语言后续 |
| 图的布局算法：力导向 vs 分层 vs 混合？ | 实际数据上对比实验 |
| 时间边延迟标签颗粒度？ | 当前两档：实时/事件驱动 和 周期 |
| 图文件格式需要二进制吗？ | JSON 够用；节点超 10 万后评估 protobuf/MessagePack |
| 运行时数据流追踪？ | 需要进程注入或动态插桩，复杂度过高，暂不碰 |
| 自然语言社区命名？ | 需要 LLM 集成，在 LLM 消费接口就绪后做 |

## 附录 A：术语表

| 术语 | 定义 |
|---|---|
| 符号节点 | 代码中定义的可命名结构（函数/类/模块/常量/接口/变量）|
| 介质节点 | 代码读写的持久化或共享资源抽象（文件/数据库/队列/缓存/网络端口/共享内存）|
| 时间节点 | 并发结构/调度器的抽象（线程/定时器/事件循环/触发条件）|
| 结构边 | 符号之间的静态代码关系（调用/继承/实现/导入/引用/实例化）|
| 数据边 | 符号对介质的 I/O 关系（读/写/订阅）|
| 时间边 | 符号在某个时间结构上执行的关系（executes_on/triggers/blocks）|
| 社区 | 耦合紧密的节点子集，由 Leiden 算法自动识别 |
| 适配器 | 针对一种语言的代码分析模块，实现统一的 LanguageAdapter 接口 |
| 全息图 | 代码库的完整节点-边图的交互式空间投影 |

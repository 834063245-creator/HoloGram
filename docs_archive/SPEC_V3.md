# SPEC V3: 约束框架 — 从 Auditor 到 Constraint Setter

> V1 定义了"要什么"。V2 定义了"什么能做到"。
> V3 定义了**"怎么看"**——当 LLM 的生产速度碾压人脑裁决带宽时，人类如何工作。
>
> 核心命题：**生产效率 > 裁决带宽 → 人类角色从 auditor 变为 constraint setter。**
> 99% 的变更自动放行。1% 触碰约束的变更路由到人。

## 1. 问题：信息流速超过人脑裁决带宽

### 1.1 现实

```
vibe coding 的生产速度（LLM）:
  一次改动 = 12 个文件 × 10 秒 = 完整 diff + 解释 + 测试结果

人脑的裁决带宽:
  审一个 diff → 30 秒到 3 分钟（取决于复杂度和阅读能力）
  审 12 个 diff → 6 到 36 分钟

比例：LLM 在 10 秒里产出的东西，人需要 6-36 分钟来裁决。
```

这还不包括对话历史、测试输出、报错信息、架构讨论——所有这些叠加在一起。

### 1.2 现有策略及其上限

vibe coder 当前的四种生存策略：

| 策略 | 做法 | 上限 |
|---|---|---|
| 信任模式 | 测试绿就合，不看 diff | 静默 bug 漏过去 |
| 减速模式 | 要求 LLM 一次只改一个文件 | LLM 速度优势消失 |
| 回滚模式 | 炸了就 reset，重来 | 不可逆破坏无法回滚 |
| 焦点模式 | 只看关键文件，其余不管 | 波及破坏漏过去 |

四种策略的本质都是**少看一点**。但如果少看的那一点恰好是关键的那一点，代价不可逆。

### 1.3 根因

```
人类角色锚定在"auditor" → 每条变更都需要人的判断 → 人脑带宽是瓶颈
```

不是信息太多。不是人太慢。是**角色错了**。

## 2. 解法：约束框架 + 例外路由

### 2.1 角色转换

```
Auditor 模式:
  每条变更 → 人看 → 人判断 → 人决定合入/拒绝
  瓶颈: 人的裁决带宽

Constraint Setter 模式:
  每条变更 → 程序自动校验约束 → 99% 放行 → 1% 触碰约束 → 路由到人
  瓶颈: 不存在。程序校验没有带宽上限。
```

### 2.2 约束框架定义

约束 = 不可逾越的边界。触碰边界 = 路由到人。完全在边界内 = 自动放行。

```
你设定的约束:
  - L4 封装穿透不能新增（容忍 0）
  - 数据流环不能新增
  - 波及节点不能超过 20
  - 公开 API 签名不能变更（除非显式允许）
  - 线程竞争窗口不能新增
  - 数值阈值变更必须路由（不管方向）

每次 LLM 改动后:
  全息仓自动运行约束校验
    → 所有约束通过 → 自动合入。你不会看到这次改动。
    → 任何一个触碰 → 变更摘要路由到你面前。
```

### 2.3 关键设计原则

1. **约束由人设定，不由程序推断。** 程序不知道"什么对你项目是重要的"。你设。
2. **约束可调。** 默认值只是起点。你可以收紧（"波及上限从 20 降到 10"）或放松（"这个模块允许 L4 穿透"）。
3. **漏判优于误判。** 宁可把安全的改动标记为不确定路由给人（误判），也不漏掉一个真正的破坏（漏判）。漏判的代价是不可逆的。误判的代价只是人多看了一眼。
4. **人不看代码，看变化形状。** 路由到人的不是 diff，是结构变化的摘要。

## 3. 破坏层级：L5-L1 信号体系

按改错的代价分级。L1 不路由给人。L5-L2 路由给人。

### L5 — 不可逆破坏

**定义：** 改了之后，git reset --hard 无法恢复原状。

| 信号 | 检测方式 | 确定性 |
|---|---|---|
| 数据库 migration 新增/变更 | 文件名匹配 `migrations/`, `*.sql`, `schema.*`, `alembic/` | ✅ |
| API 合同变更 | 公开函数签名变更（增删参数、改类型）× 该函数在 `__all__` 或 REST route 中 | ✅ |
| 序列化格式变更 | 修改了 `__getstate__`/`__setstate__`/`to_dict`/`from_dict` 或 protobuf/flatbuffer/avro schema 文件 | ✅ |
| 配置文件 key 删除/重命名 | `.yaml`/`.toml`/`.json` 配置 key 变更 × 代码中对该 key 的引用 | ✅ |

### L4 — 静默破坏

**定义：** 不报错、不崩溃、测试可能绿，但系统行为悄悄地变了。

| 信号 | 检测方式 | 确定性 |
|---|---|---|
| 数值阈值/超时/间隔变更 | AST diff: 变量名匹配 `threshold`/`timeout`/`interval`/`ttl`/`delay`/`limit` + 数值字面量变化 | ✅ |
| LLM prompt 变更 | 字符串变量名匹配 `prompt`/`system_prompt`/`template`/`instruction` + 内容变化 | ✅ |
| 排序/过滤/评分逻辑变更 | 函数名匹配 `sort`/`filter`/`rank`/`score`/`weigh` + 实现变化 | ✅ |
| 新增 L4 封装穿透 | 耦合深度计: 本次变更引入新的 `_private` 访问 | ✅ |
| 新增数据流环 | 数据流环检测: 本次变更引入新的有向环 | ✅ |

### L3 — 延迟破坏

**定义：** 改动时没问题，N 小时后行为变化。因果链在时间中断裂。

| 信号 | 检测方式 | 确定性 |
|---|---|---|
| 后台线程/定时器新增或变更 | 线程交错图: 新增 threading.Thread 创建点，或修改了 Timer/sleep 参数 | ⚠️ |
| 共享数据文件的写入逻辑变更 | 修改了操作用同一个 `*.json`/`*.db` 路径的函数 | ✅ |
| 异步任务的触发条件变更 | async 函数 × 调用它的调度器/事件循环逻辑变更 | ⚠️ |
| 节律参数变更 | 变量名匹配 `interval`/`period`/`frequency`/`cron` + 数值变化 | ✅ |

⚠️ = 保守近似，可能有漏报

### L2 — 波及破坏

**定义：** 改的地方本身没问题，但波及面超过预期。

| 信号 | 检测方式 | 确定性 |
|---|---|---|
| 波及节点数 > 阈值（默认 20） | 波及环 BFS | ✅ |
| 跨社区边新增 | 社区聚类 diff：模块 A（社区 X）新增了对模块 B（社区 Y）的 import | ✅ |
| 公开 API 签名变更 | 函数签名变更 × 该函数被 ≥ 2 个其他模块引用 | ✅ |
| 共享数据结构字段变更 | dataclass/TypedDict/model 字段增删，且该结构被 ≥ 3 个模块 import | ✅ |

### L1 — 可见破坏

**定义：** 当场崩溃/测试红。不路由给人。

**理由：** LLM 看到测试输出后会自己修。人的裁决带宽不应该浪费在"测试红了我要不要看看"上。

## 4. 变更摘要：你看到的东西

### 4.1 形态

每次 LLM 改动后，全息仓自动生成变更摘要。99% 的改动只产生一行"通过"。1% 的改动展开为以下面板：

│  改动: background/task_scheduler.py, api/schemas.py,          │
│         config/settings.py  (3 文件)                          │
│                                                              │

```

### 4.2 自动放行的条件

一次改动**全部满足**以下条件 → 自动放行，不通知人：

```
- L5 信号: 0 条
- L4 信号: 0 条
- L3 信号: 0 条
- L2 波及: ≤ 阈值
- L2 跨社区: 无
- L2 公开 API 变更: 无
```

任一项不满足 → 变更摘要路由给人。

### 4.3 局面简报：Agent 开箱即食

当变更被路由到人，人点「发送给 Agent」时，Agent 收到的不是裸的拦截信号。全息仓在生成 change summary 的同时，**预装所有已完成的 V1/V2 分析结果**，拼成一份局面简报。Agent 不再需要手动调 MCP 工具查图、查耦合、查环、查历史——程序层已经算完，直接打包。

```

│  局面简报 — 路由给 Agent                                      │
│  变更: d4e8f2a  ·  api/schemas.py:47                         │
│                                                              │

│  被依赖方 (depends on):                                      │
│    · pydantic.BaseModel                                      │
│    · api/validators.py                                       │
│  依赖方 (depended by):                                       │
│    · chat.py:102  (→ 调用 ChatRequest 构造)                   │
│    · system.py:33  (→ 调用 ChatRequest 构造)                  │
│    · mcp_server.py:18  (→ import ChatRequest)                │
│    · tests/test_api.py:55  (→ 测试构造)                       │
│                                                              │
│  ── 耦合深度（V2）────────────────────────────────────────── │
│  · api/schemas.py → api/validators.py: L1 公开 API            │

│                                                              │
│  ── 数据流环（V2）────────────────────────────────────────── │

│                                                              │

│  · 首次被路由到人 (之前 4 次均为自动放行)                      │

这个简报由 `summary.py` 的 `enrich()` 函数生成。数据全部来自已有模块：

| 信息 | 来料 |
|---|---|
| 图快照（依赖方/被依赖方） | `graph.neighbors()` + `neighbors_downstream()` |
| 耦合深度 | `coupling.py` L1-L4 分类 |
| 数据流环 | `dataflow.py` 已有环 × 本次变更节点 |
| 线程关联 | `threading.py` 冲突矩阵 × 本次变更节点 |
| 历史稳定性 | `timeline.py` SQLite 变更记录 |
| 社区关联 | `community.py` 聚类归属 |

无需新算法。所有数据都在 V1/V2 的输出里，`enrich()` 就是一通 `import` + 拼装。

### 4.4 阈值默认值

| 阈值 | 默认 | 说明 |
|---|---|---|
| L2 波及节点上限 | 20 | 可根据项目调整 |
| L2 跨社区边新增容忍 | 0 | 设为 0 = 任何跨社区边都路由 |
| L2 公开 API 签名变更容忍 | 0 | 设为 0 = 任何公开签名变更都路由 |
| L4 穿透新增容忍 | 0 | 设为 0 = 任何新增穿透都路由 |
| L4 阈值变更容忍 | 0 | 设为 0 = 任何数值阈值变更都路由 |

## 5. 约束配置：你设定的东西

### 5.1 配置文件

项目根目录 `hologram.constraints.yaml`（可选。不存在时使用默认值）：

```yaml
constraints:
  routing:
    l5_irreversible: true        # L5 永远路由，不可关闭
    l4_silent: true              # L4 默认路由
    l3_delayed: true             # L3 默认路由
    l2_blast: true               # L2 默认路由
    l1_visible: false            # L1 从不路由

  thresholds:
    blast_radius_max: 20         # 波及节点上限
    cross_community_tolerance: 0 # 跨社区边新增容忍（0 = 任何都路由）
    api_signature_tolerance: 0   # 公开 API 签名变更容忍

  allowlist:
    modules:                     # 这些模块允许 L4 穿透（不触发路由）
      - "legacy_adapter.py"
    files:                       # 这些文件的变更不触发 L3 延迟路由
      - "docs/*.md"
      - "tests/*.py"

  denylist:
    keywords:                    # 包含这些关键词的变量变更永远路由
      - "password"
      - "secret"
      - "token"
      - "api_key"
```

### 5.2 设计约束

- 配置文件简洁到"看不懂代码的人也能改 YAML 的值"
- 默认值覆盖 80% 的项目。只有特殊需求时才需要改配置
- allowlist 和 denylist 是可选的高级功能。不配也能用

## 6. 信号来源：V1/V2 哪个模块产出哪个信号

V3 不引入新的分析能力。它是对 V1/V2 分析结果的**重新组合和阈值应用**。

| 信号 | 来源模块 | V 版 |
|---|---|---|
| L5 数据库 migration | 文件名模式匹配 | V3 新增（简单规则，非算法） |
| L5 API 合同变更 | 耦合深度计 L1 + AST diff | V2 |
| L5 序列化格式变更 | AST 模式匹配 | V3 新增（简单规则） |
| L5 配置 key 删除 | AST diff + 字符串匹配 | V3 新增（简单规则） |
| L4 数值阈值/超时变更 | AST diff + 变量名模式 | V3 新增（简单规则） |
| L4 LLM prompt 变更 | 字符串 diff + 变量名模式 | V3 新增（简单规则） |
| L4 排序/过滤逻辑变更 | 函数名模式 + 实现 diff | V3 新增（简单规则） |
| L4 L4 穿透新增 | 耦合深度计（§2） | V2 |
| L4 数据流环新增 | 数据流环检测（§4） | V2 |
| L3 线程/定时器变更 | 线程交错图（§3） | V2 |
| L3 共享数据文件写逻辑变更 | 介质节点数据边 + diff | V1 + V3 |
| L3 异步触发条件变更 | 线程交错图 + AST diff | V2 + V3 |
| L3 节律参数变更 | AST diff + 变量名模式 | V3 新增（简单规则） |
| L2 波及节点数 | 波及环 BFS（V1） | V1 |
| L2 跨社区边新增 | 社区聚类 diff（V1） | V1 |
| L2 公开 API 签名变更 | 耦合深度计 L1 + AST diff | V2 |
| L2 共享数据结构字段变更 | AST diff + 引用计数 | V3 新增（简单规则） |

**V3 新增**的信号全部是简单规则——文件名模式、变量名模式、字符串匹配。不需要新算法。实现方式是在 V2 的 AST 分析流水线上附加一组模式匹配器。

## 7. 工作流：一次改动的完整生命周期

### 7.1 正常流（99%）

```
LLM 完成改动
    ↓
全息仓自动运行: python -m hologram check
    ↓
约束校验:
  L5: 0   L4: 0   L3: 0   L2: ≤阈值
    ↓
自动合入。终端输出: "✅ 通过。3 文件变更，无约束触碰。"
    ↓
人不参与。
```

### 7.2 例外流（1%）

```
LLM 完成改动
    ↓
全息仓自动运行
    ↓
约束触碰:
  L5: 1 项（API 合同变更）
  L4: 2 项（阈值变更 + 新增穿透）
    ↓
终端输出变更摘要面板（§4 形态）
星图高亮触碰约束的节点
    ↓
人看到面板:
  → 确认 → git commit
  → 拒绝 → 在终端告诉 Agent 撤回/重做
  → "Agent 先分析" → Agent 读上下文，自由回复，人再决定
    ↓
人的裁决记录写回时间轴（V1 决策时间轴已有）
    ↓
下次约束触碰时，时间轴上可见历史裁决作为上下文
```

### 7.3 Agent 的角色（重申）

```
全息仓: "这个改动触碰了 3 条约束。这是结构变化的形状。"
        ← 同时附送局面简报（§4.3），Agent 零查询直接吃
Agent:  "我已经看到了：
        - api/schemas.py 被 4 个模块依赖
        - 过去 30 天这里改过 4 次，这次是第一次被路由
        - 没有线程冲突，没有新数据流环
        我可以直接告诉你这个 API 变更是故意的还是手滑了——
        三个调用方都已经更新了。但 system.py:33 那里多传了一个参数，
        以前是允许的现在会报错。"
人:     "修 system.py:33，其他确认。"

## 9. V3 不做什么

### 9.1 不自动执行修复

约束触碰后，全息仓不告诉 Agent 怎么修。只告诉人和 Agent "什么被触碰了"。

### 9.2 不学习约束

全息仓不从人的裁决历史中自动调整约束阈值。"上次你放过了这个，下次我也放过"——不做。约束永远由人显式设定。

理由：人的裁决有上下文（"这次放过是因为赶时间，下次不一定"），程序没有这个上下文。学习会导致"某天程序替你做了你在另一个场景下不该做的决定"。

### 9.3 不覆盖运行时破坏

全息仓的静态分析无法检测以下破坏：
- 死锁（需要运行时 trace）
- 内存泄漏（需要 profiler）
- 性能退化（需要 benchmark）
- LLM 回复质量漂移（没有程序定义）

这些破坏会在运行时暴露，但它们不属于"变更摘要"的职责范围。它们属于 test suite 和 monitoring 的职责范围。全息仓不做全栈观测。全息仓做的是**代码层的破坏性评估**。

### 9.4 但为运行时工具留接口

```
全息仓产出: 结构变化摘要
Test suite: 功能正确性验证
Benchmark: 性能退化检测
Monitoring: 运行时行为漂移
Agent: 综合以上所有信号后，向人类报告
```

全息仓是信号源之一。不是唯一的。

```
